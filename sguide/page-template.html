More information can be found in the Assemble documentation, this page will only cover parameters specific to the tcom-core page templates.

Template Overview

Assemble allows for the generation of pages based on a template, meaning that only the primary body content needs to be supplied to the build task along with some data paramters to allow for the creation of a full static page. The Handlebars templates allow for a good deal of flexibility. However, you should avoid getting to elaborate if the frontend code needs to also be ported into an AEM component.

There are a few basic concepts worth understanding with regards to the templating system.

Layouts

Layouts are the base template that sets up the surrounding shell of headers, footers, navigation etc. These can be found within tcom-core/src/html/layouts. For most purposes, the responsive.hbs template is the only one that will be used. It is the base template used for telstra.com.au.

Partials

Partials are as the name suggest, partial pages or snippets of code that can be reused. They must reside within the tcom-core/src/html/partials directory (or the partial directory designated by the build task) and can be referred to by their name. Am example partial include:

{{> partial-name }}

Partials can also be fed data files in the either .yml or .json formats to allow for dynamic reusable partials. Data file directories are specified as part of the assemble.js config and can be invoked like so:

{{> partial-name data-file-name }}

Pages

Pages are the basis for all of our static content. Each individual page turns into a standalone .html output. For examples of pages, refer to tcom-core/src/html/pages.

Pages are able to take a number of data parameters that alter parts of the template or can be used to perform more advanced functionality.

Creating New Components

A new component consists of a variety of things such as markup, javascript and css. Below is the basic list of steps to go through to get a new component page setup along with associated javascript and styles, ensuring they are built into the final output.

  1. Create a new page under src/pages/components, for our example our page will be called new-component.hbs.
  2. Create associated .less file under /src/less/telstra, it should be named the same as your page - new-component.less. If required it can be organised into a folder.
  3. Add an @import for new-component.less in /src/less/telstra/telstra.less, this ensures the new styles are included in the overall less build.
  4. Create associated .js file under /src/js/telstra if required. This doesn't need to be added to a master list as the concat.js task will take care of all files in this folder and make sure they are built into the main output file.
  5. If you require any vendor libs to be included in the main output, place them into /src/js/vendor and amend /grunt/concat.js appropriately.

About Data Parameters

Data parameters are simple flags or parameters that can be passed to setup various features. At their essennce, they are variables that are built into the base templates that trigger specific code paths.

There are two main ways to pass custom data parameters; within the page header or at the assemble.js build task level.

Page Level Data

Data parameters set at the page level only affect the specific page the data is set on, this is useful for setting specific includes and values that are only required for the particular page.

Page data is set within the YAML front matter of a page, this is a header section where custom data parameters can be set for a page. Below is an example that sets the title parameter, the layout to use and an array of includes.

--- title: Page Title! layout: responsive.hbs cssIncludes: - 'include1.css' - 'include2.css' ---

Further examples can be found by opening any page under tcom-core/src/html/pages/components.

Page level data is more specific, consequently it will override any build task level data parameters that have been set.

Build Task Level Data

Data parameters can be set at the assemble.js build task level, this will insert the data paramter specified into every page that gets built as part of the task.

The example below actually shows data paramters set at two levels. The first set of parameters at the top sets responsive and corePath (prettify is a base assemble parameter). These parameters apply to all underlying build targets such as coreHtml1. Whilst the parameters basedir and globalproj only affect that specific build target. Other build targets may have other values here.

module.exports = { options: { prettify: { indent: 2 }, responsive: true, corePath: '.' }, coreHtml1: { options: { data: "./src/html/data/*.{json,yml}", partials: "./src/html/partials/**/*.hbs", layoutdir: "./src/html/layouts", helpers: [ "./src/html/helpers/*.js", "handlebars-helper-include", "handlebars-helper-rawinclude" ], basedir: 'dist', globalproj: 'global' }, files: [{ expand: true, cwd: 'src/', src: ['html/pages/components/[a-p]*.*', 'html/pages/**/components/[a-p]*.*'], flatten: true, dest: './dist/' }], }, ... other build targets }

Basic Data Parameters

Parameters are case sensitive.

Parameter Value Description
title string Sets the <title> element and the title within the white, primary navigation bar next to the logo.
h1 string Sets primary <h1> title under the navigation.
layout partial Sets the base template to use, most of the time this should be set as responsive.hbs at the build level, other templates available can be found in tcom-core/src/html/layouts.
responsive boolean Enable/disable page responsiveness, this should be set as true at the build level and false when required on a page by page basis.
cssIncludes string/array A single path or array of paths (type string) to .css files to include on the page. The includes are made after all other base .css includes.
jsIncludes string/array A single path or array of paths (type string) to .js files to include on the page. The includes are made after all other base .js includes (ie. jQuery).
theme string Page colour theme, one of blue, orange, green or magenta. Defaults to blue.
spectrum boolean Enable/disable the spectrum gradient.
page-header boolean Enables/disables the page header that contains the title and the Contact/Share buttons.
whiteBackground boolean Sets the page background as white instead of the standard grey. This is used as part of the newer style NBN Refresh pages.

Advanced Data Parameters

Parameters are case sensitive.

Advanced parameters tend to be more arbitary, allowing developers flexibilty to insert full code partials allowing for significant page changes.

Parameter Value Description
fullwidth-banner partial Allows for partial insertion into an unconstrained full width container that sits between the navigation and page header. Please refer to fullwidth container page for more information.
primaryNav partial Custom primary navigation (aka. mega menu). Whilst any parital code is accepted here, it's best to follow the existing menu structure if you wish to replicate the current mega menus. Refer to mega-menu-personal.hbs within the tcom-core/src/html/partials/header folder for an example of the telstra.com.au Personal menu.
headerBefore
headerAfter		 partial Partial include within <head> element before; cssIncludes, modernizr.js and after modernizr.js.
footerBefore
footerAfter		 partial Partial include within <body> element before; jQuery, jsIncludes and all other base .js and after.
