Telstra.com

tcom-core is a set of core templates, styles and components used to create sites and applications for Telstra. It can be used as is, precompiled or pulled into a new a project and customised as required.

  • Basic Setup
  • tcom-core as a Static Inclusion
  • tcom-core as a Dependency

Basic Setup

There are a couple of options with regards to using tcom-core.

Standalone Source

Sets up tcom-core as a fully functional build, allowing you to compile the code from source. This allows for modification of the source and exclusion of certain parts of the project being included in the output build.

  1. Download node.js v4.2.3 LTS
  2. Clone tcom-core master branch with your favourite git client (> 1.7 required if you need to manage subtrees)
  3. Open cmd, install grunt-cli globally by running npm install -g grunt-cli (requires admin)
  4. From cmd, within tcom-core directory run npm install
  5. From cmd, within tcom-core directory run grunt to start build
  6. Your default browser will open on the index page of the build

Other useful things:

  • cmder - better than windows cmd, unix style command pallette too
  • SourceTree - nice git client with integration into Stash

Compiled Distributable

A pre-compiled version of tcom-core that can be used as is.

  1. Clone tcom-core dist branch with your favourite git client.
  2. Pre-compiled files can be found in the dist folder.

tcom-core as a Static Inclusion

tcom-core can be simply included statically. This the most basic way to include tcom-core as the basis of a project and simply involves taking the pre-compiled dist branch and including the appropriate source within your project. Refer to Integration for more information.

tcom-core as a Dependency

The preferred way to use tcom-core would be as a depdency within your own repository. This allows you to import tcom-core and use it as a basis for your own build. To do so, the use of git subtree is reccommended other other methods such as submodules

subtrees are similar to the concept of submodules, whereby a child repository (tcom-core) is pulled into a parent (tcom-aem-multi). However, unlike submodules - which store a SHA commit reference - subtrees take a literal snapshot of the files and store them within the parent. Imagine it as effectively copy and pasting the child into the parent repository.

For practical purposes, this can be treated as just another folder. The benefits of the subtrees are:

  • Easier to understand, there is a lot of misunderstanding regarding how submodules work.
  • Easier to manage, as the subtree is simply a file folder, making modifications within it is the same as modifying any other file. Changes on multiple branches don't cause reference conflicts and changes can be backported to tcom-core as required in a separately managed process.
  • Isolated, as the files are snapshotted into the parent - changes within the folder are isolated to tcom-aem-multi and do no affect tcom-core. However, as the subtree is still linked to tcom-core it is possible to push and pull updates to tcom-core.

More can be found regarding subtrees here and here.

Refer to Git Subtrees and Integration for more information.
